## 1. 摘要

C 端采用 **平台内核（core）+ 特性模块（features）+ 共享层（shared）**。  
依赖方向遵循整洁架构：Presentation → Application → Domain ← Infrastructure。  
首发业务（百科地图闭环）是**第一个垂直切片**，用于验证架构，不定义架构上限。  
新业务能力通过**新 feature 模块 + 可选分包 + 组装注册**接入，不修改内核公共 API，不为未立项能力预建空页面。

---

## 2. 架构目标

| 目标 | 度量 |
|------|------|
| 可扩展 | 新业务 = 新 feature（+ 可选分包），不改 core 公共语义 |
| 可替换 | http / storage / session / state / map 实现可替换，调用方依赖端口 |
| 可测 | domain 与 application 无 `wx.*`；Node 侧单测 |
| 可运行 | 首发交付 PRD 百科闭环；主包体积可控（单包 ≤2M） |
| 可演进 | 状态库 / UI 库 / npm 有明确接入契约 |

**非目标**：未立项业务实现；空路由占位；跨端框架；重开全栈选型。

---

## 3. 软件设计原则（强制）

### 3.1 SOLID（落地）

| 原则 | 约束 |
|------|------|
| S | Page 不做 HTTP；Infrastructure 不做 Toast；Storage 不做业务规则 |
| O | 新能力用新模块 + 注册扩展；禁止 core 内 `if (feature===…)` 蔓延 |
| L | Repository / Gateway 实现可替换 |
| I | Feature 仅经 `public.ts` 外露；禁止 deep import 内部 |
| D | Domain / Application 不依赖 `wx` 与 Page |

### 3.2 分层

```text
Presentation  pages / components
Application   features/*/application（use cases）
Domain        entities / pure rules / ports
Infrastructure  http·storage·map 适配、repository 实现
```

依赖仅允许由外向内；Domain 零框架依赖。

### 3.3 其它规范

- 按**业务能力**切 feature，禁止无界 `utils` 垃圾场  
- DRY：跨 feature 复用进 `shared` 或 `core`  
- YAGNI（实现）：不写未使用抽象/空分包页；扩展缝用接口与目录纪律  
- KISS：端口 = `interface` + 工厂；不上 IoC 容器  
- 不可变更新；边界校验；错误类型显式；禁止吞错  

### 3.4 反模式黑名单

1. Page 内 `wx.request` / 散落 baseURL  
2. 巨型无界 `utils.ts`  
3. Feature 互相 deep import  
4. Domain 引用 `wx` / 组件  
5. 无类型 `globalData` 总线  
6. 为未立项业务注册空页面  
7. 登录成功自动续办写操作  
8. Secret 进小程序包  
9. 非 GCJ-02 坐标直入 `<map>`  
10. 无门面隔离的重型全局状态「先引着」

## 4. 架构风格

```text
app.ts / app.json     组装根：hydrate、注册已启用 feature
    ├── core/         内核端口与基建（无业务语义）
    ├── shared/       与业务无关的 UI / lib
    └── features/*    业务模块（可独立增减）
```

| 区域 | 职责 | 可否依赖 feature |
|------|------|------------------|
| core | http、session、storage、state、map、config、navigation、error | 否 |
| shared | 通用展示与纯工具 | 否 |
| features/\<name\> | 单一业务能力的完整切片 | 仅经他 feature 的 public |
| app 组装根 | 启用模块、路由、preload | 可依赖 public |

**扩展标准路径**

1. 新建 `features/<name>/`（标准模板）  
2. 导出 `public.ts`  
3. 需要时独立 subpackage  
4. 在 app 组装处注册路由/入口  
5. 不改其他 feature 内部；协作抽 core 端口或 shared 契约  

**首发启用模块**：`features/encyclopedia`、`features/account`。  
不预建其它 feature 目录。

## 5. 技术栈

| 层 | 选型 | 升级触发 |
|----|------|----------|
| 语言 | TypeScript strict | — |
| 运行时 | 微信原生，基础库 3.x+ | 跟官方 |
| 组件 | glass-easel + Component | — |
| 注入 | lazyCodeLoading: requiredComponents | — |
| 渲染 | 默认 WebView；Skyline 能力门禁后开 | 真机矩阵通过 |
| 样式 | WXSS + CSS 变量令牌 | 设计系统膨胀 |
| 状态 | Store Port + 内存/storage 实现 | 跨 feature 订阅爆炸 → 换实现 |
| 网络 | core/http 封装 wx.request | — |
| 存储 | core/storage 版本化 key | — |
| 地图 | core/map 端口 + 微信适配 | 聚合/视野策略替换适配器 |
| 测试 | Vitest（纯模块） | — |
| Lint | ESLint + Prettier + tsc | — |
| npm | 默认零业务运行时依赖 | feature 显式引入 |

### 5.1 第三方接入契约

| 接入 | 契约 |
|------|------|
| 状态库 | 只换 `core/state` 或 store **实现**；对外 `getState/setState/subscribe` 稳定 |
| UI 库 | 进 `shared/ui` 或 feature presentation；业务组件包装第三方 |
| 工具 npm | 经 core/shared 再导出；Page 不直接堆依赖 |

### 6.1 Feature 模板

```text
features/<name>/
  domain/          entities, rules, ports
  application/     use cases
  infrastructure/  port 实现
  presentation/    pages, components
  public.ts        唯一外露 API
```

**依赖规则**

- domain → 无 wx；仅类型与纯逻辑  
- application → domain + ports  
- infrastructure → ports + core  
- presentation → application + shared  
- 跨 feature → 仅 `public.ts`  

### 6.2 分包

| 包 | 内容 |
|----|------|
| 主包 | Shell + Tab（map/mine）+ 需分享冷启的 detail + core/shared + 启动所需 presentation |
| package-account | login、history、favorites、about、agreement、privacy 等低频路径 |
| 未来 | 按**新 feature 边界**增分包，不预注册空包 |

## 7. 核心子系统

### 7.1 HTTP

- 唯一 API 出口；解析 `{ success, data, error, meta }`  
- `HttpError { status, code, message }`  
- Bearer 注入；401 → session.clear；**不**在 http 层弹登录  
- 写操作不重试；与 server envelope 对齐  

### 7.2 Session

- Port：`hydrate | getToken | setSession | clear | subscribe | isLoggedIn`  
- 登录 UI 在 account；session 实现在 core  
- `ensureAuthenticated`：引导登录后**默认不自动重放**原写操作（PRD）  

### 7.3 Storage

- Key：`cq/v1/<scope>/...`；可迁移  
- 业务规则在 domain（如浏览 MRU 上限 200）  

### 7.4 State

```ts
interface Store<S> {
  getState(): S
  setState(reducer: (prev: S) => S): void
  subscribe(listener: () => void): () => void
}
```

### 7.5 Map Port

```ts
interface MapMarkerVm {
  id: string
  lng: number // GCJ-02
  lat: number
  styleKey: string
  selected?: boolean
}
interface MapViewport {
  includePoints?: Array<{ lng: number; lat: number }>
  center?: { lng: number; lat: number }
  scale?: number
}
```

业务 feature 负责「数据 → MarkerVm[]」；内核不包含具体玩法语义。

### 7.6 Config

- `env`：多环境 `API_BASE_URL`（禁止默认提交个人局域网 IP）  
- `capabilities`：能力开关读取接口；远程配置可后挂  

### 7.7 Navigation / 组装根

- `core/navigation` 封装跳转与 returnUrl  
- `app.ts` 为 composition root：创建依赖、hydrate session  

### 7.8 Error

- 基础设施错误 → 应用错误 → UI 文案策略  
- 禁止 infrastructure 内 Toast

## 9. 通用扩展机制

| 类型 | 做法 |
|------|------|
| 新业务 | feature 模板 + public + 注册 |

## 附录 A：升级触发条件

| 现象 | 动作 |
|------|------|
| 跨页共享订阅复杂、手写 store 成本高 | 替换 core/state 实现（如 mobx-miniprogram） |
| 设计系统组件爆炸 | shared/ui 引入并包装组件库 |
| 单 feature 主包超标 | 该 feature 独立分包 |
| 点位规模使全量 setData 不可行 | 替换 encyclopedia 的加载策略 / map 适配 |
| 表单校验极复杂 | 该 feature 局部引入 schema 库 |

## 附录 B：假想 Feature X 接入清单（演练）

1. `features/x/{domain,application,infrastructure,presentation,public.ts}`  
2. 需要网络则 repository 用 `core/http`  
3. 需要登录则调 `account/public` 的 ensureAuthenticated  
4. 需要地图则产出 `MapMarkerVm[]`  
5. `app.json` 注册页面或 subpackage  
6. `app.ts` 组装依赖  
7. 不修改 encyclopedia 内部文件
